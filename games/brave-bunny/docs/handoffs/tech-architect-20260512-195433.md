## Chosen option: **B — `WeaponArchetypeConfig` sidecar SO,
polymorphic via the existing `[BraveRegister]` registry.**

Rationale (2 sentences): Option A/C are junk-drawer patterns that
conflate per-weapon constants (`arm_time_ms`) with per-level stats
(`damage`) and clutter every weapon's inspector with 6+ irrelevant
fields. Option B mirrors ADR-0009 exactly — one concrete subclass per
archetype variant, type-name string keyed via `[BraveRegister]`,
inspector shows only relevant fields, importer dispatches cleanly on
JSON archetype + key-presence disambiguator.

## Non-obvious constraints discovered while reading existing code

1. **Archetype JSON is coarser than the C# subclasses need to be.** The
   JSON `archetype` field has 5 values (`projectile`, `area`, `aura`,
   `summon`, `utility-beam`), but `area` covers Daisy Mine (mine),
   Thunder Cloud (cloud), and Cob Mortar (splash). The importer must
   disambiguate by **key-presence**, not just by the archetype string.
   Documented in §Consequences §"Importer".
2. **`WaveSpawner.cs:47-50` is the only switch-table call site for
   `EnemyRole` in production code.** A second reference lives in
   `EnemyDefinition.OnValidate` but it's a `!= Swarmer` check that
   handles `Boss` correctly without modification. Two importer file
   references are just string-mapping (`ApplyEnumByJsonString`).
3. **Archetype fields are *per-weapon*, not *per-level*.** Per-level
   overrides exist only as perks (L3 `arm_time_ms`, L4
   `cloud_lifetime_ms`). This is why putting them in `WeaponLevelData`
   was the wrong shape — the data wants a `weapon.archetypeConfig`
   reference with parallel per-level arrays *inside* the config SO,
   not a sibling of `damage`/`fireRate`.
4. **`projectilePrefab` is already nullable on `WeaponDefinition`** —
   the precedent for archetype-conditional fields exists. Adding a
   nullable `archetypeConfig?` is shape-consistent.
